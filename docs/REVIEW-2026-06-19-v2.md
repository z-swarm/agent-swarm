### 3.2 NEW-1 (P1) agent_review EVAL 规则过粗

**位置**:`tools/agent_review.py:147`(修复前)

```python
{
    "category": "EVAL",
    "pattern": re.compile(r"\b(eval|exec)\s*\("),
}
```

**问题**:
- 误报率极高:`asyncio.get_event_loop()` / `conn.execute()` / 注释 `// don't use eval()` / 测试 fixture 中的 `assert eval(...)` 都会被命中
- 没有 file extension 白名单,`.md`/`.json`/`.yaml`/`.toml` 里的 `eval(` 文字也被扫
- 也没有路径黑名单,`.venv/` / `node_modules/` 第三方库中调用被命中
- **影响**:dogfooding 工具自用价值打折;CI 接入后会被频繁误报淹没真问题

**修复**(`48fd9c2`):
- 新增 `_SOURCE_EXTENSIONS` 白名单(`.py`/`.js`/`.ts`/`.go`/`.rs`/`.java` 等)
- 新增 `_PATH_SKIP_SUBSTRINGS` 黑名单(`.venv/` / `node_modules/` / `vendor/` / `.git/` / 缓存目录)
- EVAL 规则收紧为 `(?<![a-zA-Z0-9_."'])\b(eval|exec)\s*\(`:排除 `self.eval()` 方法调用和词边界(`evaluate` / `developer` / `execution`)
- 新增 `_line_is_string_literal` 启发式:match 之前未配对引号为奇数 ⇒ 在字符串里,应忽略

**已知 limitation**:`f"{eval(x)}"` 这种 f-string 内的 eval 会被误跳。要彻底解决需 AST 分析,regex 不可达。

### 3.3 NEW-2 (P2) WebSocketSink 慢消费者背压测试覆盖薄

**位置**:`tests/unit/test_websocket_sink.py:159-185`(v1)

```python
async def test_websocket_sink_enqueue_drops_when_queue_full() -> None:
    # 只验证 _enqueue 丢最旧,没验证 _send_loop 在网络慢时的行为
    client.queue = asyncio.Queue(maxsize=2)
    await sink._enqueue(client, {"x": 1})
    ...
```

**问题**:
- 单测只验证 `_enqueue` 入队时丢最旧,**没有验证实际 `_send_loop` 在网络慢时的行为**
- 当 `ws.send_json()` 慢于 emit 速度时,`dropped_events` 会不会持续增长?`consume()` 会不会被阻塞?
- e2e 测试也没有"故意让 consumer 慢"的场景

**修复**(`48fd9c2`):
- 新增 `test_websocket_sink_slow_consumer_does_not_block_consume`:验证 100 次 `consume()` 在 1s 内完成 + 慢客户端 dropped ≥ 90
- 新增 `test_websocket_sink_fast_consumer_unaffected_by_slow_peer`:验证慢消费者丢消息不影响快消费者(多客户端场景)

## 6. 结论(v2)

**整体评价**:项目工程化程度在 Python 多 agent 框架中属于**上等偏上水平**。v2(W10-W13)新增了 4 个完整的 Weekly Slice,每个都达到"实现 + 单测 + e2e + 文档 + 验收脚本"5 件套水准,工程化收尾成熟度维持。

**v2 期间没有引入新 P0 风险**;唯一发现的是 2 个 P1/P2 工具/测试细节问题,均已闭环。

**最值得借鉴的做法**(v2 强化):
- W10 飞书连接器:`${VAR}` SecretManager 引用 + HMAC 签名验证是"真安全"的范本
- W11 ChannelApprover:用 IM 卡片做异步审批是 UX + 安全双赢的设计
- W12 WebSocketSink:多 sink fan-out + 背压丢最旧是实时仪表盘的可靠基础
- W13 dogfooding:用项目自己 Swarm 审项目 PR 是工程文化的高阶形态

**最值得关注的短板**(v2 仍有):
- f-string 内的 `eval()` 用 regex 检测不到(W12-NEW-1 已知 limitation);要彻底解决需 AST 分析
- 5 元组索引在 session 量极大(>100M events)时,SQLite 单文件性能会成瓶颈;远期需切到 ClickHouse / DuckDB
- 飞书连接器目前只接 Lark,钉钉/企业微信适配可作为未来 W14+ 方向
