## 2. 强项(做得好的地方)

### 2.1 设计文档质量极高

- `DESIGN.md` 2 400+ 行,**每个决策都写"为什么"**(不是只写"是什么")
- 版本迭代痕迹清晰(v3 → v4 → v4.1 → v4.2,每版都列修订摘要)
- 反模式会被显式记录并否决:
  - §6.6 拒绝 KB+CC 合并方案,3 条硬约束逐条解释
  - §6.4 解释"为什么不需要悲观锁"
  - §7.1 合并 v3 的 `mode / allowed_actions / risk_profile` 为单一 `AgentCapabilities`
- §17 工程实践与交付门禁(v4.2 新增)让"做对、做完、能跑"有可量化标准

### 2.2 安全防护是真的,不是 PPT

`sandbox.py` 4 道防线:
1. **命令白名单** —— argv[0] 精确匹配(只读型命令,拒绝 python/awk/sed/find/node 等图灵完备/全盘搜索类)
2. **shell 元字符检测** —— `;& | $ < > \ ! # * ? [ ] { } ( ) ' "` 出现即拒绝
3. **`shlex.split` + `shell=False`** —— 防命令注入
4. **`realpath` 验证** —— 防 symlink + 绝对路径逃逸 + TOCTOU

`policy.py` 默认 secure-by-default:
- `read_file`: 敏感路径(`/etc/passwd`、`~/.ssh/`、`*.pem` 等 30+ 条)→ DENY
- `run_command`: 命中命令黑名单 → DENY;HIGH 风险默认 REQUIRE_APPROVAL
- `write_file`: 路径不在白名单 → DENY

`tests/security/test_attack_suite.py` 25 条攻击用例覆盖。

### 2.3 测试金字塔完整

```
tests/
├── unit/         34 个单测文件(每个核心模块都有专门测试)
├── e2e/           9 个 Weekly Slice DoD 端到端验证
├── golden/       10 个 P1 Golden Case(G-001..G-010)+ 5 个 P2 根因命中率用例
├── security/     25 条攻击用例

```

W7-W9 每个切片都有 e2e + DoD 文档 + 验收脚本(`tools/verify_w7_dod.py` 等)三件套。

### 2.4 协议抽象干净

- `CollaborationProtocol` 基类(刻意改名避开 stdlib `typing.Protocol`,§6.3 注释)
- 两个协议落地:
  - `DelegateMode`(Lead + Worker 委派)
  - `AdversarialVerifier`(对抗式验证,4 条收敛优先级 + 5 种兜底)
- Swarm 通过 `set_protocol()` + `run_with_protocol()` 注册执行
- `JudgeFn = Callable[[Agent, str, int], Awaitable[Judgement]]` 设计让单测可以用确定性脚本回放,避免绑死 LLM

### 2.5 错误兜底到位

- `AdversarialVerifier`: judge_fn 抛异常 → 该 (agent, hypothesis) 转 UNCERTAIN,不污染其他判断(§6.2.5)
- `AdversarialVerifier._round_all_failed`: 无存活假设时不再误判 stall(H2 修复)
- `MCPRegistry`: `risk_overrides` 非 dict → fail-fast(H1 修复)
- `StdioMCPClient`: 重连后清空 `_pending` + 取消旧 future(H3 修复)
- `MCPClient`: stderr drain 后台 task 防止 pipe buffer 撑爆(M1 修复)

### 2.6 Commit 历史组织良好

- 每周切片有明确编号(W1-W9)和 DoD 标签
- `review: W7-W9 审查 fix-up` 这种自我审查提交表明团队有自检文化

## 3. 风险点(需要关注)

### P0 — 紧迫(本次审查没发现)

最近一次自我审查(`90c8dfa`)已修复 3 个 H 危 + 4 个 M 危,本轮未发现新的 P0 问题。

### P1 — 高优先级(需要尽快处理)

#### 3.1 MCP 工具是否走 SecurityPolicy 校验

- `security/policy.py` 的 `ToolRisk` 等级是给 builtin 工具用的

- **建议**:补一条 e2e 测试,断言 MCP 工具调用前 `policy.check_tool()` 被调用,`risk_overrides` 真的生效

#### 3.2 CLI 不支持 Anthropic

- README §Quickstart 只提 `OPENAI_API_KEY`
- `cli/main.py:110` 把 `--api-key` 强写 `os.environ["OPENAI_API_KEY"]`,没考虑 provider 分发

- **建议**:加 `--provider` 选项 + 自动读 `ANTHROPIC_API_KEY` env,否则"Anthropic 支持"是空头支票

#### 3.3 mypy 2.1.0 内部错误

- 项目 `mypy src/` 直接 internal error
- 可能是 dataclass + `from __future__ import annotations` 在 strict 模式下的兼容问题
- **影响**:IDE 类型提示质量下降;CI 无法用 mypy 卡类型
- **建议**:`mypy --show-traceback` 看具体堆栈,可能是 `protocols.py` 的 Protocol 类或 `types.py` 的 dataclass 触发

### P2 — 中优先级(应该处理)

#### 3.4 ApprovalFlow 还是最小占位

- `policy.py` 写"W6+ 计划:ApprovalFlow 集成 REQUIRE_APPROVAL 决策"

### 7.3 逐项验证

#### P1-3.1 MCP 工具 SecurityPolicy 双闸门 ✅

`src/agent_swarm/mcp/adapter.py:60-92` 实现防御深度的两道闸门:

```python
# 闸门 1:SecurityPolicy(如注入)
if self.policy is not None:
    decision = self.policy.check_tool(self.name, arguments)
    if decision.decision == "DENY":
        return f"[error] policy denied: {decision.reason}"
    if decision.decision == "REQUIRE_APPROVAL":
        return f"[error] requires approval: ..."

# 闸门 2:risk 二次校验(防通用 policy 不知道 MCP 工具存在)
risk_level = _RISK_ORDER.get(self.risk, 1)
if risk_level >= _RISK_ORDER["high"]:
    return f"[error] requires approval: tool {self.name} has risk={self.risk} ..."
```

`verify_p1_dod.py` 输出:
```
[P1-3.1] MCP 工具 SecurityPolicy 链路 + risk_overrides 二次闸门
  ✓ 8 个新单测通过(policy ALLOW/DENY/REQUIRE_APPROVAL + risk 二次闸门)
```

**细节**:拒绝时返回字符串 `[error] ...` 而不是抛异常,是 graceful denial 设计,让 LLM 能识别并放弃。建议未来在 agent system prompt 文档里明说 `[error]` 前缀语义。
